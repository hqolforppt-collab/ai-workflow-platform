## 1. Design principles (additions to v2.0's six)

7. **One gate check, everywhere.** Governance enforcement lives in exactly one shared module; every mutating path ‚Äî CLI or MCP ‚Äî imports it. Duplicated safety logic is how G1 happened.
8. **Fail closed.** Missing or unreadable gate state means *refuse*, never "warn and deploy". The only ungated path is a dry-run that routes to `validate_bpmn`.
9. **No feature without a test.** Every workstream below ships with its acceptance test in the same change. A feature whose test doesn't exist is DRAFT, not done. (`--aggregate` shipped broken because nothing ever executed it.)
10. **Spec and tree must agree.** Where the audit found the tree diverging *usefully* from the plan (17 rules, extra tier globs), the spec is corrected to match the tree ‚Äî not the other way around.

### W1 ‚Äî Governance unification (closes G1; corrects P4)

**New shared package: `packages/awp-governance/`** (workspace-local, zero deps beyond `yaml`):

```
packages/awp-governance/
‚îú‚îÄ‚îÄ package.json                # name: @awp/governance, type: module
‚îî‚îÄ‚îÄ src/
    ‚îú‚îÄ‚îÄ gates.js                # checkGates(repoRoot, blueprintId) ‚Üí {approved, missing[], unapproved[]}
    ‚îî‚îÄ‚îÄ index.js                # re-exports
```

Contract:

- `checkGates` reads `.governance/gates/G{1..4}-*.yaml`; **all four must exist and have `status: approved`** for the given blueprint id. Missing file ‚áí `unapproved`. No warn-and-continue path.
- Consumed by **both** `packages/awp-cli/src/commands/flowable.js` (replacing its private `checkGates`, removing the fail-open branch) and `packages/flowable-mcp-server` (new).

**Blueprint-id plumbing (the P4 fix):**

- `awp flowable convert` already stamps `deploy-manifest.json`; it now must include `blueprint_id` (read from the staged files' `_meta.blueprint_id`, or derived `BLU-<SLUG>` for monoliths).
- MCP `deploy` gains an optional `blueprint_id` param; when omitted it reads it from a `deploy-manifest.json` passed among `files`. If no blueprint id can be resolved ‚áí **refuse** (cannot gate-check what it cannot identify).
- MCP `start_process` / `complete_task` operate on already-deployed definitions; they gate on the deployment's recorded blueprint id when available, and are otherwise allowed only against definitions deployed through a gate-checked path (recorded in `deploy-report.json`). Simplest enforceable rule for v2.1: `start_process`/`complete_task` require `AWP_ALLOW_RUNTIME_MUTATION=1` **or** a resolvable approved blueprint id ‚Äî explicit, documented, fail-closed.

Acceptance:
- Unit: ungated blueprint ‚Üí `deploy` (CLI and MCP) returns a refusal naming the unapproved gates; approved fixture ‚Üí proceeds.
- The fail-open branch is gone: absent `.governance/gates/` ‚áí refusal.

### W2 ‚Äî Correctness fixes (closes G4, G5, G9-partial)

**W2.1 Fix `--aggregate` (G4).** Rewrite `mergeStagedFiles` in `build.js`:
- ESM imports only (delete the `require()` calls).
- Parse each `0N-*.yaml` with `yaml`, strip `_meta`, and perform the ¬ß7.5 merge: shallow section union; `workflows = { ...root-scope entries, dependent: [...] }` merged per scope; **assert id uniqueness** across merged sections (reusing VAL-020 patterns) and fail with the colliding ids listed.
- Output a **single-document** `blueprint.yaml` with all 28 sections.

**W2.2 Expose the rule engine on the CLI (G5).** New routing in `index.js` / `commands/validate.js`:
- `awp validate` (no args) ‚Üí existing platform-OS validation (unchanged, back-compat).
- `awp validate <path>` where `<path>` is a blueprint dir (contains `0N-*.yaml`) or a `blueprint.yaml` ‚Üí run `src/validate.js`, print the ¬ß5.4 report format (`16/17`-style tallies, failed rule IDs + hints), exit non-zero on failure.
- Help text updated (also fixes the missing `--aggregate` in usage ‚Äî low gap).

**W2.3 Implement VAL-032; classify VAL-040 (G9-partial).**
- VAL-032 (dangling refs): collect every id defined across sections; every `ref:`/`*-id` field must resolve. Real check, not `passed.push`.
- VAL-040 (PII flagged): keep informational, but *say so* in output (`VAL-040 ‚ìò informational`), not a silent pass.

**W2.4 `model.js` resilience (audit enhancement).** Add bounded retry with exponential backoff for 429/5xx/network errors (3 attempts), and surface a structured stage-failure event so `--resume-from` is the documented recovery path.

### W3 ‚Äî MCP delivery (closes G2, G7, G14; corrects P2)

Plus a "Connecting the Flowable MCP server" section in `packages/flowable-mcp-server/README.md` (new) covering Claude Code, opencode, and generic stdio clients. `.mcp.json` is added to the v2.1 file manifest (P2 corrected).

**W3.2 Modularize the server (G14).** Split the monolith into the originally planned layout ‚Äî `src/client.js` (REST + auth + error mapping) and `src/tools/*.js` (one file per tool, each exporting `{name, description, schema, handler}`) ‚Äî with `src/index.js` reduced to registration. This is what makes W4's per-tool unit tests possible. `deploy.js`, `start-process.js`, `complete-task.js` import `@awp/governance` (W1).

**W3.3 Skill delegation (G7).** Add the ¬ß8.5 `mcp:` blocks to the five skills:

```yaml

  server: flowable
  verify-with: validate_bpmn      # get_form for form-modeling; list_processes for root-model-selection
  when: after-generation
```

Acceptance: `.skills/registry.yaml` lint (extend `.tools/validate-spec.sh`) asserts the five skills carry an `mcp:` block referencing a tool that exists in the server.

### W5 ‚Äî Converter depth & golden example (closes G6, G8)

**W5.1 Converter (G6).** `flowable.js` conversion must, for both staged and monolithic input:
- emit one `<key>.bpmn20.xml` per workflow ‚Äî **key from the workflow's own id/slug**, never a generic `root`;

Acceptance: converting the login-registration golden example yields **‚â•5 artifacts** (root + 2 dependent BPMN, ‚â•1 form.json, ‚â•1 dmn) and `awp flowable validate` passes against Docker Flowable in CI.

## 5. Risks

| Risk | Sev | Mitigation |
|---|---|---|
| Fail-closed gating breaks existing ad-hoc deploy flows | Med | Dry-run (`validate_bpmn`) stays ungated; refusal message names exactly which gate files to approve; documented in README. |
| MCP runtime tools (start/complete) over-blocked by strict gating | Med | Explicit `AWP_ALLOW_RUNTIME_MUTATION=1` escape hatch, documented, off by default ‚Äî visible in tool output when used. |
| Monolith‚Üímodular MCP refactor regresses tool behavior | Low | W4 unit tests written against the *current* tool contracts first, then refactor under green tests. |
| Aggregate merge semantics surprise downstream consumers | Low | Golden test asserts 28-section single-doc shape; `--aggregate` output validated by the rule engine before write. |
| Docker Flowable flakiness in CI | Med | Integration job isolated (existing `bpmn-roundtrip.yml` pattern); unit + mock E2E never depend on the engine. |

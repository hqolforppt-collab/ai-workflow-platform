## 1. Executive Summary

`/workflow-builder` has strong spec design but no execution loop: today a user must run `awp build "story"` to *assemble a prompt file*, then manually `cat …prompt.md | <model-cli> > blueprint.yaml`. There is no model call, no progress, no Flowable path, no cheap-model story, and output is one 1,200-line YAML.

v2.0 delivers, additively:

| # | Enhancement | One-line outcome |
|---|---|---|
| **E1** | One-command Claude Code build | `/workflow-builder "story"` runs end-to-end and writes files — no manual piping. |
| **E2** | Improved UX | Streamed `[STAGE n/6]` progress, per-stage partial writes, `--resume-from`, structured coverage report. |
| **E3** | Flowable MCP integration | `packages/flowable-mcp-server` (8 tools) + `awp flowable convert/deploy` turn blueprints into deployed BPMN/CMMN/DMN/Forms. |
| **E4** | Cheap-model compatibility | Tiered prompts (`small`/`medium`/`large`), a ~800-token schema summary, per-stage section slicing, and **deterministic** (non-LLM) validation. |
| **E5** | Staged 6-file YAML output | 6 incremental files with a dependency graph + optional aggregated `blueprint.yaml`. |

All five are **additive and independently shippable** — Phase 1 improves UX with zero new deps; Phase 3 (MCP) is optional and gated behind a running Flowable engine.

## 3. Design Principles (constraints on every decision below)

1. **Additive, never destructive.** `awp build "story"` with no new flags behaves exactly as v0.1. New behavior is opt-in via flags; the Claude Code adapter opts in by default but always has an inline fallback.
2. **Prompt.md stays canonical.** All model input is assembled from `.commands/workflow-builder/prompt.md` + schema + KB. The programmatic path and the inline path share the same source of truth.
3. **Determinism where correctness matters.** Validation and tier/section slicing are code, not model judgment. LLMs generate; code checks.
4. **Two backends, one contract.** Whether generation runs via `awp build --execute` (programmatic API call) or inline in Claude Code, the *stage marker format*, *file layout*, and *validation* are identical.
5. **Flowable-native vocabulary.** `process-definitions`, `case-definitions`, `decision-tables`, `form-definitions`, `deployments`, `process-instances`, `tasks` (VAL-013 domains + Requirement 4).
6. **No secrets in the tree** (Constitution C5). Config references env vars only; `.awp/` is git-ignored.

## 4. Enhancement 1 — One-Command Claude Code Build

### 4.1 Target experience

```
> /workflow-builder "Create Login and Registration"
[STAGE 1/6] Initialized      — story parsed: 3 actors, 5 capabilities, tier=medium, L6
[STAGE 2/6] In Progress      — 32 domains (4 explicit, 28 hidden), 87 hidden reqs
[STAGE 3/6] Generating App   — actors/roles/data-model/master-data/knowledge
[STAGE 4/6] Root Models      — BPMN root process + security/api/events/integrations
[STAGE 5/6] Dependent Models — email-verification, password-reset + audit/logging/monitoring
[STAGE 6/6] Forms            — 8 form-definitions + testing/deployment/ops/compliance
✓ blueprints/login-and-registration/  (6 files, 1,230 lines)
✓ Validation: 16/16 rules passed (deterministic)
✓ Flowable-ready — deploy with:  awp flowable deploy blueprints/login-and-registration/
```

### 4.2 Three-layer architecture

```
Layer 1  .claude/commands/workflow-builder.md   (adapter — REVISED, additive)
           ├─ if `awp` on PATH → shell out: awp build "$ARGS" --execute --staged
           └─ else → INLINE fallback: run prompt.md across the 6 stages,
                     emitting the same [STAGE n/6] markers itself
                                   │
Layer 2  packages/awp-cli/src/commands/build.js  (REVISED — new flags, back-compat default)
           awp build "story"                  → v0.1 behavior (assemble .prompt.md)   [unchanged]
           awp build "story" --execute        → call model, write output
           awp build "story" --execute --staged → 6-file staged generation (§7)
                                   │
Layer 3  packages/awp-cli/src/model.js  (NEW — provider abstraction)
           providers: anthropic | openai | ollama
           config:    .awp/config.yaml (model id, tier, env-var key names)
           streaming: async iterator yields {stage, status, summary} → Layer 2 prints markers
```

### 4.3 Adapter revision (`.claude/commands/workflow-builder.md`)

Keep the existing frontmatter and canonical-source pointer. Append an execution preamble:

```md
## Execution mode (v2.0)

1. If an `awp` CLI is available in this environment, run:
     awp build "$ARGUMENTS" --execute --staged
   Stream its `[STAGE n/6]` lines to the user, then show the coverage report.
2. Otherwise, execute inline: run the 6-step pipeline yourself, but WRITE ONE FILE
   PER STAGE to `blueprints/<slug>/0N-<name>.yaml` (see stages.yaml), printing a
   `[STAGE n/6] <name> — <summary>` line as each file is written. Then run the
   deterministic validator conceptually (schema + cross-refs) and print the report.
```

The inline path guarantees the *same UX and file layout* even with no CLI installed — this is the fallback that removes lock-in.

### 4.5 CLI surface (final flag contract)

`parseFlags` (in `index.js`) already supports `--flag` (boolean) and `--flag=value`. No parser change needed. New/confirmed flags on `build`:

| Flag | Type | Default | Meaning |
|---|---|---|---|
| `--execute` | bool | false | Call the model; without it, assemble `.prompt.md` (v0.1). |
| `--staged` | bool | false (CLI) / true (adapter) | Emit 6 files vs. one `blueprint.yaml`. |
| `--level=L1..L6` | str | L6 | Maturity (existing). |
| `--model-tier=small\|medium\|large` | str | auto | Force a tier; else derived from config `model.id`. |
| `--out=<path>` | str | `blueprints/<slug>/` (staged) or `blueprints/<slug>.yaml` | Output target. |
| `--resume-from=stage-0N` | str | — | Re-run one stage using prior stage files as context (§5.3). |
| `--no-validate` | bool | false | Skip deterministic validation (debug only). |
| `--check` | bool | false | Existing dry-run (inputs-present check). |

## 5. Enhancement 2 — Improved UX

### 5.1 Pain → fix map

| Pain | Fix |
|---|---|
| Silent generation | `[STAGE n/6] <Name> — <summary>` streamed as each stage completes |
| Monolithic output | 6 navigable files (§7) |
| All-or-nothing failure | each stage writes its own file; failure at N preserves 1..N-1 |
| Dense prompt | tiered variants + per-stage section slicing (§6) |
| Opaque validation | deterministic validator prints failing **rule IDs + fix hints** |
| Hidden cross-cutting | coverage report links each hidden req to `source: discovery/<domain>/<rule>` |

### 5.2 Stage marker contract (stable, parseable)

```
[STAGE <n>/6] <Name> — <summary>
```

`<Name>` ∈ {`Initialized`, `In Progress`, `Generating App`, `Root Models`, `Dependent Models`, `Forms`}. Emitted by **both** backends. Machine-parseable prefix `^\[STAGE (\d)/6\] `; human-readable tail. `model.js` yields these as structured events `{stage:n, name, status, summary}`; `build.js` renders them; the inline adapter prints them directly.

### 5.3 Partial recovery / resume

Each stage writes atomically to its own file before the next begins. On failure at stage N:

```bash
awp build "story" --execute --staged --resume-from=stage-05 \
  --out=blueprints/login-and-registration/
```

`--resume-from=stage-05` reads `01..04-*.yaml` from `--out`, loads them as context, and regenerates only stage 5 (and downstream 6 if present). Tier and level are re-read from config. Idempotent: re-running a completed stage overwrites its file deterministically (same inputs → same slice).

### 5.4 Coverage report (final format)

```
─────────────────────────────────────────────────────────

─────────────────────────────────────────────────────────
Story:      "Create Login and Registration"
Maturity:   L6 (Enterprise)      Tier: medium (claude-sonnet-5)

Validation: 16/16 rules passed ✓   (deterministic)

─────────────────────────────────────────────────────────
blueprints/login-and-registration/
  01-initialized.yaml       120 ln  project, domains(explicit), tier
  02-in-progress.yaml       210 ln  requirements, risks, assumptions
  03-generating-app.yaml    380 ln  actors, roles, data-model, master-data, knowledge
  04-root-models.yaml       150 ln  root workflow, security, api, events, integrations
  05-dependent-models.yaml  200 ln  sub-processes, audit, logging, monitoring, notifications
  06-forms.yaml             170 ln  forms, pages, testing, deployment, ops, docs, gov, compliance
  Total: 1,230 lines / 6 files
─────────────────────────────────────────────────────────
Flowable ready ✓   awp flowable deploy blueprints/login-and-registration/
─────────────────────────────────────────────────────────
```

On failure the same report prints with `Validation: 13/16 ✗` and a block:

```
FAILED RULES
  VAL-021 acceptance-criteria-required  04-root-models.yaml  workflows[reset-pw].steps[verify]
          → add >=1 Given/When/Then acceptance criterion
  VAL-013 auth-story-mandatory-domains  01-initialized.yaml  missing: disaster-recovery
          → activate domain via implication closure; do not drop silently
Status: DRAFT-INVALID (files retained for repair; re-run with --resume-from)
```

## 6. Enhancement 4 — Cheap-Model Compatibility (placed before E3 because E5 depends on it)

### 6.1 Tiers (`.commands/workflow-builder/tiers.yaml`, NEW)

```yaml
id: workflow-builder-tiers
version: 1.0.0
default_tier: medium
tiers:
  - id: small
    max_context_tokens: 8000
    match:  [gpt-3.5*, llama*8b*, "claude-haiku*", mistral*7b*]
    schema: .schemas/workflow-blueprint/schema-summary.yaml   # ~800 tokens
    strategy: per-stage-sliced          # only current stage's sections in-prompt
    max_maturity: L3
  - id: medium
    max_context_tokens: 32000
    match:  ["gpt-4o-mini", "claude-sonnet*", "gpt-4.1-mini"]
    schema: .schemas/workflow-blueprint/schema.yaml
    strategy: per-stage-sliced
    max_maturity: L6
  - id: large
    max_context_tokens: 100000
    match:  ["gpt-4o", "claude-opus*", "claude-fable*"]
    schema: .schemas/workflow-blueprint/schema.yaml
    strategy: staged-or-monolithic      # may do all sections in one call
    max_maturity: L6
```

Tier resolution order: `--model-tier` flag → `.awp/config.yaml tier_override` → glob match of `model.id` against `tiers[].match` → `default_tier`.

### 6.2 Schema summary (`.schemas/workflow-blueprint/schema-summary.yaml`, NEW)

Condensed contract for the `small` tier: section names + required field names + types only; validation rules referenced by ID, no prose. Target ≤ 1,000 tokens (vs. ~4,500 for full schema). Generated once by a helper `awp schema summarize` (Phase 2 stretch) or hand-authored; kept in sync via a golden test that asserts every `required: true` field in `schema.yaml` appears in the summary.

### 6.3 Per-stage section slicing

Instead of one prompt with all 28 sections, each stage prompt carries only:

- the stage's `schema_sections` (from `stages.yaml`, §7.2),
- the relevant KB slice (`trigger-map` at stage 2; `implication-map` for active domains at stages 4–5),
- prior stage outputs as compact context (IDs + summaries, not full YAML, for the `small` tier).

Token budget per stage (medium tier, est.): prompt ≈ 2.5k–4k, output ≈ 1k–2k — comfortably inside 8k for even `small` on L1–L3.

### 6.4 Deterministic validation (`packages/awp-cli/src/validate.js`, NEW)

The 16 rules become **machine-checkable code**, independent of the model. Mapping:

| Rule class | Rules | Check implementation |
|---|---|---|
| Structural | VAL-001, VAL-002 | all 28 keys present across staged files; always-populated sections non-empty & not `not-applicable` |
| Hidden-coverage | VAL-010, VAL-011 | `domains.cross-cutting.length ≥ 12`; `requirements.hidden` non-empty |
| Traceability | VAL-012 | every hidden item has `source: discovery/<domain>[/<rule>]` matching a KB domain |
| Auth mandatory | VAL-013 | if story matches `login\|registration\|signin\|auth\|password\|account`, assert the 25 named domains present |
| Actionability | VAL-020..VAL-024 | id-pattern regexes; each requirement has ≥1 G/W/T AC; each API ≥2 responses incl. non-2xx; each form field has `validation`+`error-message`; no `TBD/TODO` |
| Ops/compliance | VAL-030..VAL-042 | per-rule field presence for testing/deployment/operations/governance/compliance |

Implemented with `yaml` (already a dep) + `zod` (new) for shape assertions and hand-written cross-reference checks. Returns `{passed:[], failed:[{rule, file, path, hint}]}`. Exposed as `awp validate blueprints/<slug>/` and called automatically at end of stage 6 (and after any `--resume-from`).

### 6.5 Cost comparison

| | v0.1 monolithic | v2.0 staged + tiered |
|---|---|---|
| Model | Opus only | GPT-3.5/Haiku (L1–L3) · Sonnet (L4–L6) |
| Prompt tokens | ~8,200 | ~2.5k (L1–L3) + ~4k (L4–L6) per stage |
| Output tokens | ~15,000 | ~5k (S1–S3) + ~10k (S4–S6) |
| Fits 8k context | ✗ | ✓ per stage |
| Est. cost | ~$0.50 | ~$0.08 |

## 7. Enhancement 5 — Staged 6-File YAML Output

### 7.1 The 6 stages and the exact 28-section mapping

The 28 schema sections (verified in `schema.yaml`) are partitioned so **each section is authored in exactly one stage**. `workflows` is the only section split (root vs. dependent), merged at aggregation.

| Stage | File | Sections authored | Pipeline steps | Depends on |
|---|---|---|---|---|
| 1 Initialized | `01-initialized.yaml` | `project`, `domains` (explicit + tier + maturity) | parse-story | — |
| 2 In Progress | `02-in-progress.yaml` | `requirements`, `risks`, `assumptions` (+ resolved constraints) | classify-domains, expand-hidden, resolve-constraints | 01 |
| 3 Generating App | `03-generating-app.yaml` | `actors`, `roles`, `data-model`, `master-data`, `knowledge` | populate-schema | 02 |
| 4 Root Models | `04-root-models.yaml` | `workflows`(root), `security`, `api`, `events`, `integrations` | populate-schema | 03 |
| 5 Dependent Models | `05-dependent-models.yaml` | `workflows`(sub/dependent), `notifications`, `monitoring`, `logging`, `audit` | populate-schema | 04 |
| 6 Forms | `06-forms.yaml` | `forms`, `pages`, `testing`, `deployment`, `operations`, `support`, `documentation`, `governance`, `compliance` | populate-schema, emit-and-validate | 04, 05 |

Section count check: 2 + 3 + 5 + 5 + 4 + 9 = 28 (with `workflows` counted once, split across S4/S5). ✔ All 28 covered, none duplicated.

`domains.cross-cutting` (the hidden set, VAL-010) is finalized in stage 2 alongside `requirements.hidden`, so both hidden-coverage rules are checkable by the end of stage 2.

### 7.2 Stage config (`.commands/workflow-builder/stages.yaml`, NEW)

```yaml
id: workflow-builder-stages
version: 1.0.0
output_layout: directory        # blueprints/<slug>/0N-*.yaml
aggregate_file: blueprint.yaml   # optional monolith, built by `awp build --aggregate`

stages:
  - id: stage-01
    name: Initialized
    file: 01-initialized.yaml
    pipeline_steps: [parse-story]
    schema_sections: [project, domains]
    depends_on: []
  - id: stage-02
    name: In Progress
    file: 02-in-progress.yaml
    pipeline_steps: [classify-domains, expand-hidden, resolve-constraints]
    schema_sections: [requirements, risks, assumptions]
    kb_slice: [trigger-map, implication-map]
    depends_on: [stage-01]
  - id: stage-03
    name: Generating App
    file: 03-generating-app.yaml
    pipeline_steps: [populate-schema]
    schema_sections: [actors, roles, data-model, master-data, knowledge]
    depends_on: [stage-02]
  - id: stage-04
    name: Root Models
    file: 04-root-models.yaml
    pipeline_steps: [populate-schema]
    schema_sections: [workflows, security, api, events, integrations]
    workflows_scope: root
    depends_on: [stage-03]
  - id: stage-05
    name: Dependent Models
    file: 05-dependent-models.yaml
    pipeline_steps: [populate-schema]
    schema_sections: [workflows, notifications, monitoring, logging, audit]
    workflows_scope: dependent
    depends_on: [stage-04]
  - id: stage-06
    name: Forms
    file: 06-forms.yaml
    pipeline_steps: [populate-schema, emit-and-validate]
    schema_sections: [forms, pages, testing, deployment, operations, support, documentation, governance, compliance]
    depends_on: [stage-04, stage-05]
```

### 7.3 Dependency graph (strictly linear — honest about ordering)

```
01 ─▶ 02 ─▶ 03 ─▶ 04 ─▶ 05 ─▶ 06
                        └──────────┘  06 also reads 04
```

**Correction vs. earlier drafts:** stages 5 and 6 are **not** run in parallel. Stage 6 authors `forms`, which reference user-tasks defined in *both* the root workflow (04) and dependent workflows (05); it cannot start until 05 is complete. The chain is strictly sequential. (A future optimization could pre-draft form skeletons from 04 while 05 runs, but that is out of scope for v2.0 to keep the merge deterministic and avoid the race called out in the earlier plan's risk table.)

### 7.4 Per-file envelope (traceability, VAL-012)

Every staged file begins with a trace/meta block so partial files are self-describing and audit-clean:

```yaml
# 04-root-models.yaml
_meta:
  blueprint_id: BLU-LOGIN-AND-REGISTRATION
  stage: stage-04
  produced_by: workflow-builder@2.0.0
  tier: medium
  maturity_level: L6
  depends_on: [stage-03]
  trace:
    requirement_source: 02-in-progress.yaml
    upstream: [01-initialized.yaml, 02-in-progress.yaml, 03-generating-app.yaml]
workflows:
  root:
    id: WF-LOGIN-ROOT
    ...
```

### 7.5 Aggregation & backward compatibility

- Default CLI (`awp build "story"` no flags): unchanged v0.1 (assemble `.prompt.md`).
- `awp build "story" --execute` (no `--staged`): single `blueprints/<slug>.yaml` (28 sections, one file) — the classic monolith, now actually model-generated.
- `awp build "story" --execute --staged`: 6 files (Claude Code adapter default).
- `awp build --aggregate blueprints/<slug>/`: deterministically merges 6 files → `blueprints/<slug>/blueprint.yaml` for consumers that still expect one file. Merge = shallow section union; `workflows` = `{...root, ...dependent}`.

### 8.2 Tool signatures (8 tools — concrete params & returns)

All params/returns validated with `zod`. REST endpoints are Flowable v6/v7 REST API.

```ts
// flowable.health_check  — GET /service/management/engine
health_check(): { ok: boolean, engine: string, version: string, url: string }

// flowable.deploy  — POST /repository/deployments (multipart)
deploy(input: {
  files: { name: string, content: string }[]   // *.bpmn20.xml | *.cmmn.xml | *.dmn | *.form.json
  deployment_name?: string
  tenant_id?: string
}): {
  deployment_id: string
  process_definitions: { id: string, key: string, version: number }[]
  case_definitions:    { id: string, key: string, version: number }[]
  decision_tables:     { id: string, key: string, version: number }[]
  form_definitions:    { id: string, key: string }[]
}

// flowable.validate_bpmn  — dry-run: deploy then delete (cascade), return parse errors
validate_bpmn(input: {
  files: { name: string, content: string }[]
}): { valid: boolean, errors: { file: string, line?: number, message: string }[] }

// flowable.list_processes  — GET /repository/process-definitions
list_processes(input?: {
  key?: string, latest?: boolean, tenant_id?: string
}): { total: number, data: { id: string, key: string, name: string, version: number }[] }

// flowable.start_process  — POST /runtime/process-instances
start_process(input: {
  process_definition_key: string
  business_key?: string
  variables?: Record<string, unknown>
  tenant_id?: string
}): { instance_id: string, definition_id: string, ended: boolean }

// flowable.query_tasks  — GET /runtime/tasks
query_tasks(input?: {
  process_instance_id?: string, assignee?: string, candidate_group?: string
}): { total: number, data: { id: string, name: string, assignee?: string, form_key?: string }[] }

// flowable.complete_task  — POST /runtime/tasks/{taskId}  (action=complete)
complete_task(input: {
  task_id: string
  variables?: Record<string, unknown>
}): { completed: boolean, task_id: string }

// flowable.get_form  — GET /form/form-definitions or /runtime/tasks/{id}/form-data
get_form(input: {
  form_key?: string, task_id?: string
}): { key: string, fields: { id: string, type: string, required: boolean }[] }
```

**Safety:** `deploy`, `start_process`, and `complete_task` mutate a live engine. Per Constitution R2, these are **blocked until gates G1–G4 are approved** for the blueprint being deployed — `deploy.js` checks `.governance/gates/` state for the blueprint id and refuses if any of G1–G4 is unapproved (unless `--dry-run`, which routes to `validate_bpmn`). `validate_bpmn`, `health_check`, `list_processes`, `query_tasks`, `get_form` are read/parse-only and always allowed.

### 8.3 YAML → Flowable converter: `awp flowable convert`

New CLI command group `flowable` (routed in `index.js` `switch`). `convert` reads staged blueprint files and emits Flowable-native artifacts:

```
awp flowable convert blueprints/<slug>/  [--out=blueprints/<slug>/flowable/]

  04-root-models.yaml  workflows.root            → <key>.bpmn20.xml  (or .cmmn.xml if root-model=case)
  05-dependent-models.yaml workflows.dependent[] → <key>.bpmn20.xml each
  05-dependent-models.yaml dmn/decision-tables[] → <key>.dmn
  06-forms.yaml        forms[]                    → <key>.form.json
  → writes blueprints/<slug>/flowable/*  + deploy-manifest.json
```

Conversion reuses the transform logic proven in `scripts/bpmn-roundtrip.mjs`. `root-model-decision` (from stage 4) selects BPMN vs. CMMN.

### 8.4 `awp flowable deploy` / `validate`

```
awp flowable validate blueprints/<slug>/     # convert (in-mem) → flowable.validate_bpmn → report; no engine mutation
awp flowable deploy   blueprints/<slug>/      # gate-check → convert → flowable.deploy → deploy-report.json
```

## 11. Risks & Mitigations

| Risk | Sev | Prob | Mitigation |
|---|---|---|---|
| Staged output breaks monolith consumers | Med | Med | `--aggregate` regenerates single `blueprint.yaml`; default CLI behavior unchanged |
| Flowable engine not running | High | High | `health_check` reports connectivity; `convert`/`validate` work with **no engine**; `deploy` is the only engine-required path; support Docker + Flowable Cloud REST |
| Cheap model emits invalid YAML | Med | Med | deterministic `validate.js` runs post-generation; invalid → structured `DRAFT-INVALID` + rule IDs, never silent pass |
| Section split (`workflows` in S4+S5) causes id collisions | Med | Low | `workflows_scope: root\|dependent` namespaces ids; `--aggregate` merge asserts unique ids (VAL-020) |
| Extra API calls raise latency | Low | High | tokens roughly flat; sequential stages add wall-clock but cut cost via cheaper per-tier models; `large` tier may run monolithic |
| Adapter change breaks existing users | Med | Low | inline fallback preserves current behavior; `--execute` is additive |
| Mutating MCP tools deploy un-gated models | High | Low | `deploy.js` checks `.governance/gates/` G1–G4 before any deploy; read-only tools always allowed (R2) |
| `.awp/config.yaml` leaks secrets | High | Low | file stores env-var **names** only; `.awp/` git-ignored (C5) |

# /workflow-builder v3.0 — Generalization Plan

**Date:** 2026-07-05
**Status:** PROPOSED
**Baseline:** `main` @ cfb40c0 (v2.1 gap remediation merged, PR #14)
**Predecessors:** `.hermes/plans/2026-07-04-final-claude-architecture.md` (v2.0), `.hermes/plans/2026-07-04-v2.1-gap-remediation-plan.md`

---

## 1. Why v3.0

The v2.x evaluation verdict: **architecture strong (8/10), delivery on the stated
target partial (6/10)**. The engine — trigger-map discovery, implication fixpoint,
staged 6-file output, 17-rule deterministic validation, fail-closed governance —
is built and tested. What limits the platform is *content breadth and matching
intelligence*:

1. **Knowledge library is auth-centric.** 26 domains, ~13 lines each. A story
   about invoice approval or inventory activates only the generic baseline.
2. **Trigger-map keywords already promise domains that don't exist.**
   `payment, checkout, billing, invoice → [fraud-detection, audit, compliance]`
   has no `payments` domain file; `upload, file, document → [security, backup]`
   has no `file-management` domain file.
3. **Keyword matching is a lookup, not understanding.** Synonyms outside the
   trigger list are silently missed.
4. **`schema.yaml` is not machine-parseable** (`list[string]` pseudo-types), so
   the 28-section contract is enforced only partially by the 17 rules.
5. **One golden example, in the same domain the KB is tuned for.** No evidence
   the pipeline generalizes.
6. **Practices are static** (v1.0.0, no staleness detection); validation checks
   structure, not substance.

v3.0's single goal: **make "one line in → best-practice blueprint out" true for
any business workflow, not just auth** — without weakening the deterministic,
traceable core that makes v2.x trustworthy.

## 2. Workstreams

### W1 — Domain Library ×4: from 26 to ~100 domains, organized as packs

**Gap addressed:** #1, #2 (thin, auth-biased KB; orphan triggers)

The KB architecture already makes this a pure content problem: a domain is a
small YAML file with `triggers`, `implies`, `standards`, `constraints`,
`requirement-seeds`, `blueprint-sections`. No code changes needed to add one.

**W1.1 — Formalize the domain-file contract first.**
- Write `.schemas/domain-knowledge/schema.yaml` (strict YAML this time — see W3)
  defining required fields, ID conventions (`<DOM>-C<n>` constraints,
  `discovery/<domain>/REQ-<n>` seeds), and the rule that every `implies` target
  must exist in the registry.
- Add `awp validate --kb` mode: every domain file parses, conforms, and the
  implication graph is closed (no dangling `implies`, no orphan trigger-map
  domains). This immediately flags the existing `payments` / `file-management`
  orphans.
- Add `.memory/domain-knowledge/TEMPLATE.yaml` + `AUTHORING.md` (how to write a
  domain: cite a standard, concrete numbers, 2–4 constraints, 1–3 seeds).

**W1.2 — Ship domain packs.** Group new domains under
`.memory/domain-knowledge/<pack>/` with the index gaining a `packs:` registry.
Priority order (chosen by how often each appears in enterprise Flowable work):

| Pack | Domains (examples) | Anchor standards |
|------|--------------------|------------------|
| `payments` | payments, refunds, invoicing, tax, reconciliation, pricing, subscription-billing | PCI-DSS 4.0, PSD2/SCA, SOX |
| `approvals` | approval-chains, delegation, escalation, four-eyes, sla-management | ISO 9001, COSO |
| `documents` | file-management, document-lifecycle, e-signature, retention, versioning | eIDAS, ISO 15489, GDPR Art.17 |
| `hr` | onboarding, offboarding, leave-management, performance-review, payroll-handoff | SOC 2, local labor-law hooks |
| `commerce` | ordering, inventory, fulfillment, shipping, returns, catalog | GS1, Incoterms |
| `data` | etl-pipelines, data-quality, master-data-sync, archiving, reporting-pipeline | DAMA-DMBOK, GDPR Art.5 |
| `integration` | api-gateway, webhooks, message-queues, idempotency, retry-policy, circuit-breaker | (patterns: EIP, saga, outbox) |
| `scheduling` | calendars, reminders, recurring-jobs, timers, business-hours | RFC 5545 |

Target: **8 packs, ~70 new domains** (~13–20 lines each, same density as the
auth set). Each pack lands as its own PR with its own trigger-map and
implication-map additions.

**W1.3 — Rebalance the trigger map.** Move `trigger-map` and `implication-map`
entries into the pack files (index becomes an aggregator built by
`awp kb build-index`, checked in CI for drift like `--aggregate` is today).
Prevents the index from becoming a 1,000-line merge-conflict magnet.

**Acceptance:**
- `awp validate --kb` passes: 0 dangling implies, 0 orphan trigger domains.
- Every existing VAL rule still passes on the auth golden example (additive-only).
- A story from each pack activates ≥ 1 functional domain from that pack plus
  the baseline (new golden tests, one per pack — see W4).

**Effort:** L (largest workstream; content-heavy, parallelizable per pack).

### W2 — Classification Engine v2: two-phase matching

**Gap addressed:** #3 (keyword lookup misses synonyms; LLM bridges gaps invisibly)

Keep determinism as the *floor*, add intelligence as a *proposer that must be
verified* — never let the LLM silently invent domains.

**W2.1 — Synonym-expanded triggers (deterministic).**
- Extend the domain-file schema: `triggers` entries may be
  `{term, synonyms: [...]}`. Matching normalizes case/stemming (`registering`
  → `register`) in a new `packages/awp-cli/src/classify.js` with unit tests.
- CLI: `awp classify "<story>"` prints activated domains + which keyword/synonym
  fired — making Stage 2 auditable standalone.

**W2.2 — LLM proposer, registry-verified (the two-phase step).**
- Pipeline step `classify-domains` becomes: (a) deterministic match as today;
  (b) model proposes additional candidate domains **only from the registry
  list** (closed vocabulary — the prompt embeds the domain ids); (c) every
  proposal is tagged `source: proposed/<model>` and listed under a new
  `assumptions` entry so a human can strike it. Proposals never satisfy
  VAL-013-style minimums on their own; deterministic matches do.
- Negation handling moves into the engine: `classify.js` recognizes
  `no <term>` / `without <term>` / `<term>-less` patterns and emits
  `not-applicable (user-negated)` entries deterministically, instead of
  trusting the model to honor negations.

**W2.3 — Fallback beacon.** When a story matches *zero* functional domains,
the coverage report must say so explicitly
(`Domains: baseline-only — no functional pack matched; consider authoring one`)
rather than silently producing a generic blueprint. This turns misses into KB
backlog items.

**Acceptance:**
- `classify.js` unit tests: synonyms, stemming, negation, zero-match beacon.
- Golden test: a story phrased with synonyms only (no exact trigger keyword)
  still activates the right pack deterministically.
- A proposed-but-unverifiable domain never appears as `explicit` or `hidden`
  without the `proposed/` source tag.

**Effort:** M. Depends on W1.1 (schema for synonyms).

### W3 — Schema Formalization: make the 28-section contract machine-true

**Gap addressed:** #4 (`schema.yaml` unparseable; conformance unchecked)

**W3.1 — Rewrite `.schemas/workflow-blueprint/schema.yaml` as strict YAML.**
Replace pseudo-types (`type: list[string]`) with a real, parseable convention:

```yaml
fields:
  - name: steps
    type: list
    items: string
    required: true
```

Keep the file human-readable (it doubles as documentation) but require
`YAML.parse` to succeed — add that as a unit test on day one.

**W3.2 — Generate, don't hand-maintain.**
- `schema-summary.yaml` (the small-model variant) becomes generated:
  `awp schema summarize` writes it from `schema.yaml`; CI diffs like
  `--aggregate` today. Kills the drift class permanently.
- The `.schemas/` directory joins `awp validate` platform-OS `OS_DIRS` (found
  during v2.1: it was never spec-checked).

**W3.3 — Full-conformance rule family VAL-050..VAL-054.**
- VAL-050: every blueprint section's fields conform to schema types.
- VAL-051: no undeclared sections/fields (additive extensions go under `x-`).
- VAL-052: required fields present per maturity level (L-gated requiredness).
- VAL-053: ID formats (`REQ-\d{3}`, `WF-\d{3}`, `DEC-\d{3}`, `FRM-\d{3}`).
- VAL-054: `_meta.depends_on` stages exist and are acyclic.
- Rule count message: keep the "N rules" strings *generated* from
  `validation-rules.yaml` in adapters (sync-adapters emits the count) so the
  16→17 drift bug class can't recur at 22.

**Acceptance:** schema parses; summary generated + CI-diffed; golden example
passes 22/22; a fixture with a type violation fails VAL-050 with a pointing
error message.

**Effort:** M. Independent of W1/W2 — can start immediately.

### W4 — Golden Examples ×3: prove generalization

**Gap addressed:** #5 (one example, same domain as the KB bias)

**W4.1 — `examples/workflow-builder/invoice-approval/`** (payments + approvals
packs). Deliberately **DMN-heavy**: approval-routing decision table
(amount/department/vendor-risk → approver chain), four-eyes rule, SLA
escalation timer. Exercises: DMN converter, `approvals` pack, escalation
sub-workflow.

**W4.2 — `examples/workflow-builder/employee-onboarding/`** (hr + documents
packs). Deliberately **CMMN-shaped**: case with discretionary tasks (equipment,
badge, training), document collection with retention rules, e-signature
integration. Exercises: CMMN root-model selection (the `root-model-selection`
skill currently has no CMMN-selecting example anywhere), `documents` pack.

**W4.3 — Each example ships the full v2.1 contract:** 6 staged files +
aggregated `blueprint.yaml` (CI-diffed), `flowable/` converted artifacts,
`golden-tests.yaml`, README. CI runs validate + aggregate-diff + convert on
**all** examples (matrix, not hardcoded to login-registration).

**W4.4 — Coverage-breadth golden tests.** One `GT-` test per pack (8 total):
`"story X activates pack Y domains"` — cheap, deterministic (runs
`awp classify`), keeps packs honest as they evolve.

**Acceptance:** 3 examples × (staged 22/22, aggregate 22/22, convert emits
BPMN/CMMN/DMN/forms as appropriate); CI matrix green; CMMN path of the
converter gains its first real regression test.

**Effort:** M–L. Depends on W1 (packs) and W3 (rule count). The two examples
can be built in parallel.

### W5 — Knowledge Freshness & Provenance

**Gap addressed:** #6a (static practices, no staleness detection)

**W5.1 — Provenance fields in the domain schema (W1.1):**

```yaml
standards:
  - { id: NIST-800-63B, version: "4", verified: 2026-07-01, url: "https://..." }
review:
  cadence: P12M          # ISO-8601 period
  last-reviewed: 2026-07-01
  owner: security-wg
```

**W5.2 — Staleness lint.** `awp validate --kb` warns when
`last-reviewed + cadence < today`; CI emits the warning list as a job summary
(warn, not fail — content review is human work, per the constitution's
human-approval principle).

**W5.3 — Review workflow.** `.governance/` gains a lightweight
`KB-review` transition doc: quarterly issue auto-filed (GitHub Action on cron)
listing stale domains; review = PR bumping `last-reviewed` with a
one-line changelog per touched constraint. Keeps the KB living without
inventing new infrastructure.

**Acceptance:** all ~100 domain files carry provenance; lint flags a
deliberately-stale fixture; cron workflow exists and is documented.

**Effort:** S. Rides on W1.1's schema.

### W6 — Substance Validation: beyond structure

**Gap addressed:** #6b (17/22 rules can pass on mediocre content)

Structure rules can't judge quality, but they can catch the *symptoms* of
low-effort content deterministically — and an optional graded pass can do the
rest without ever gating.

**W6.1 — Deterministic substance lints VAL-060..VAL-063 (hard rules):**
- VAL-060: no placeholder smells — `TBD|TODO|FIXME|<.*>|xxx` in any value.
- VAL-061: acceptance criteria are real Given/When/Then (all three clauses,
  each ≥ 3 words, measurable token present — number/duration/percent — for
  performance-class requirements).
- VAL-062: numeric plausibility windows (session timeout 5m–24h, rate limits
  1–10 000/min, retention 1d–10y) — catches copy-paste nonsense values.
- VAL-063: duplicated-text detector — no two requirements/steps share > 85%
  identical text (cheap token-shingle similarity, no model needed).

**W6.2 — Optional graded review (never gates).** `awp review <blueprint-dir>`:
sends each section + its activated domains' constraints to the configured
model with a rubric (specificity, internal consistency, constraint coverage),
emits `review-report.yaml` with per-section scores + findings tagged
`advisory`. Uses the existing model.js provider stack (incl. `mock` for CI
smoke). Explicitly *not* wired into gates — consistent with "humans approve,
models prepare evidence".

**W6.3 — Coverage scoring in the report.** The coverage report gains a
`Constraint coverage: <n>/<m> domain constraints reflected in blueprint`
line — computed deterministically (constraint id cited somewhere in the
blueprint's `source`/`trace` fields). Makes "did the blueprint actually use
the best practices?" a visible number.

**Acceptance:** fixtures for each VAL-06x (pass + fail); `awp review` runs
E2E under mock provider in CI; coverage line appears in golden examples'
reports at 100%.

**Effort:** M. Independent; VAL-06x can land before W1 finishes.

## 3. Sequencing & dependencies

```
Phase A (parallel):  W1.1 domain schema ──► W1.2 packs (8 PRs, parallel)
                     W3 schema formalization
                     W6.1 substance lints
Phase B:             W2 classification v2   (needs W1.1)
                     W5 provenance          (needs W1.1)
Phase C:             W4 golden examples     (needs W1.2 packs + W3 rule count)
                     W6.2/6.3 graded review + coverage score
Phase D:             docs/spec sync (counts regenerated, README, CHANGELOG 3.0.0)
```

Suggested increments: `v3.0-alpha` = Phase A, `v3.0-beta` = +B, `v3.0` = +C/D.
Each pack PR is independently shippable — no big-bang merge.

| Metric | v2.1 today | v3.0 target |
|--------|-----------|-------------|
| Functional domains | 6 (all identity-adjacent) | ~40 across 8 packs |
| Total domains | 26 | ~100 |
| Orphan trigger-map domains | 2 (payments, file-mgmt) | 0 (CI-enforced) |
| Golden examples | 1 (auth) | 3 (auth, DMN-heavy, CMMN-heavy) |

| Constraint-coverage visibility | none | % line in every report |

## 5. Risks

- **Content quality at volume** — 70 new domains is where quality could dilute.
  Mitigation: AUTHORING.md + per-pack review by the graded rubric (W6.2) before
  merge; packs land one PR at a time.
- **Trigger collisions across packs** (e.g. "order" in commerce vs. approvals).
  Mitigation: `awp validate --kb` gains a collision check — same keyword mapping
  to > 2 packs requires an explicit disambiguation entry.
- **Plausibility windows (VAL-062) false-positives** on unusual-but-valid specs.
  Mitigation: windows are warnings at L1–L3, hard at L4+ only; override via
  documented `x-justification` field.
- **Scope creep toward a rules engine.** The constitution's line holds: models
  propose, deterministic rules verify, humans approve. W6.2 is advisory forever.
